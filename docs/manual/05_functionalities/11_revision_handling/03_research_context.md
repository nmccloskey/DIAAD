# Revision Handling Research Context

Revision handling is part of research validity, not only project housekeeping. In discourse analysis, later coding work often reveals transcription problems, metadata mistakes, or ambiguous analytic units. DIAAD supports iterative work, but it cannot decide whether a revision changes the meaning of already-completed coding.

## Why Revisions Matter

Manual coding decisions are usually tied to a specific transcript row, sample, or utterance. If a transcript row changes after coding, the earlier code may no longer describe the same evidence. If an identifier changes, the code may no longer be joinable to the transcript row at all.

This is why transcript-table review is placed before coding-file generation in the recommended workflow.

## Not All Revisions Are Equal

Some corrections are clerical. Others are analytic. For example:

- correcting a misspelled participant group label may affect grouping but not utterance coding;
- changing an utterance from unintelligible to intelligible may affect word counts, target vocabulary coverage, and manual codes;
- splitting one utterance into two may affect utterance-level reliability and any code tied to row identity.

The project team must decide when a revision requires recoding. DIAAD can preserve files and rerun commands, but it does not replace that methodological judgment.

## Audit Value

Timestamped output directories, effective configuration files, and run manifests help preserve the history of a workflow. They are most useful when paired with explicit revision notes about why a transcript table or coding file changed.
