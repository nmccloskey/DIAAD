# `examples` Usage Guide

Use `diaad examples` before running unfamiliar commands on study data. The generated files are synthetic and are meant to make DIAAD's file contracts tangible.

## Full Example Dataset

Run:

```bash
diaad examples --config config
```

In a normal CLI run, DIAAD creates a timestamped output directory and writes:

```text
example_files_full_dataset/
  README.md
  config/
  input/
  expected_outputs/
  example_manifest.json
```

The full package is comprehensive. It pools synthetic inputs and expected outputs for the covered command set. Its `expected_outputs/` directory is organized as an atlas by module and command, so it is easier to browse across DIAAD than to treat as a single normal workflow output tree.

## Command-Specific Packages

Run:

```bash
diaad examples --for-command "transcripts tabularize" --config config
```

The output package uses:

```text
example_files_transcripts_tabularize/
  README.md
  example_config/
  example_input/
  example_output/
  example_logs/
  example_manifest.json
```

Command-specific packages are designed as small toy datasets that more directly reflect expected use cases for one command or a small command set. The `example_` prefixes help keep synthetic files distinct from real project files.

To request multiple commands in one command-specific package, repeat the flag:

```bash
diaad examples --for-command "cus analyze" --for-command "cus evaluate" --config config
```

Do not put multiple commands inside one `--for-command` value. The CLI expects one canonical command per flag.

`templates subset` must currently be generated by itself because that command expects exactly one Excel workbook in its input directory.

## Configuration Behavior

The `--config` argument controls the DIAAD run wrapper and output location. The contents of the generated example package come from DIAAD's packaged example specs.

Each package includes runnable config files. Full packages use:

```text
config/project.yaml
config/advanced.yaml
```

Command-specific packages use:

```text
example_config/project.yaml
example_config/advanced.yaml
```

These example configs are not a verbatim copy of DIAAD's installed defaults. They include demonstration settings for synthetic metadata, reliability fraction, coder counts, bin counts, blinding, and custom target-vocabulary resources. See Configuration (`docs/manual/02_operation/04_configuration.md`) for the current DIAAD defaults.

## Web App Generation

In the web app, examples are generated from Part 3: Commands.

If no commands are selected, click:

```text
Download full example dataset
```

If commands are selected, click:

```text
Download example files for selected commands
```

The downloaded ZIP contains the same kind of full or command-specific package that the CLI can generate.

## Relationship To Example I/O Docs

Generated Example I/O pages are manual views, not the same files as the downloadable packages. They show readable command examples, snippets, preview tables, and runtime-shaped paths.

The downloadable packages are runnable toy file sets. They may not match the generated documentation examples byte-for-byte, and that is acceptable. Both support the same goal: helping users understand DIAAD project structure and command behavior before working with real data.

## Force

Use `--force` only when an existing example package should be replaced:

```bash
diaad examples --for-command "vocab check" --force --config config
```

With ordinary timestamped CLI runs, output folders are usually new, so `--force` is most relevant when writing into a reused destination or when generating examples through lower-level tooling.

## Common Problems

If `--for-command` fails, confirm that the command is a canonical DIAAD command such as `cus files`, `vocab check`, or `turns analyze`.

If the generated package does not resemble a normal project, check whether you generated the full dataset instead of a command-specific package. The full dataset is an atlas; command-specific packages are closer to toy projects.

If generated outputs contain filled coding values, remember that those values are synthetic and are included so downstream commands can be demonstrated.
