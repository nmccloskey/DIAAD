# Example Dataset and Command-Specific Packages Usage Guide

Generated examples are a low-risk way to learn DIAAD's file contracts. Use them as a tour before creating, uploading, or editing real project files.

## Use The Full Dataset As An Atlas

The full dataset package is comprehensive:

```bash
diaad examples
```

It writes:

```text
example_files_full_dataset/
  README.md
  config/
  input/
  expected_outputs/
  example_manifest.json
```

The `expected_outputs/` tree is organized by module and command. That makes it good for browsing, but it is not the exact output layout of one ordinary DIAAD run.

## Use Command-Specific Packages As Toy Projects

When you are preparing to run one command, generate that command's example package:

```bash
diaad examples --for-command "vocab check"
```

The package uses:

```text
example_config/
example_input/
example_output/
example_logs/
```

Start with the README, then inspect the config and input files. The `example_output/` folder shows representative output shape. The `example_manifest.json` gives a precise index of artifacts and runtime-shaped display paths.

## Combine Related Commands

Some command sets make sense together:

```bash
diaad examples --for-command "cus analyze" --for-command "cus evaluate"
```

Each `--for-command` value should be one canonical command. Do not place a comma-separated command list inside one flag.

Combined command packages can reuse prior outputs as later inputs. That is intentional: it makes the package look more like a realistic small workflow.

`templates subset` is a special case and should currently be generated by itself.

## Understand The Configuration Layers

There are two configurations to keep separate:

- the configuration used by the `diaad examples` run to decide where to write the package;
- the runnable example configuration inside the generated package.

Example package configs are demonstration configs generated from packaged specs. They are not a verbatim copy of DIAAD's installed defaults.

## Compare Examples To Manual Pages

Read generated Example I/O when you want concise manual previews. Use downloadable packages when you want to inspect files directly.

The generated documentation and packages may differ in specific snippets, but they are aligned by command identity. Both are meant to help users understand DIAAD before working with real data.

## Move From Example To Project

After inspecting an example:

1. copy the project pattern, not the synthetic data;
2. adapt your real `project.yaml` and `advanced.yaml`;
3. run a dry-run config check;
4. run the command on a small real subset if appropriate;
5. inspect outputs before continuing.
